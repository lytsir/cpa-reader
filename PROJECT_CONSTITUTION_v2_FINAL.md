## 0. 核心铁律（不可违反）

| 铁律 | 内容 | 违反后果 |
|------|------|---------|
| **原文零改动** | 中间栏加载的必须是完整原始HTML。切分只能插入不可见的 `section-anchor` 锚点。**Phase 2 允许修正因 PDF 转 HTML 导致的：(1) 标题标签层级错误（如 h1→h3），(2) 明显文字格式错误（如序号缺失、错行、段落断裂）。除此之外，严禁修改原文文字、表格结构、图片、公式。** | 项目失败，全部回滚 |

| **统一框架+动态加载** | 最终产品为一个 `index.html` 三栏框架，各科原文和大白话作为外部资源按需加载。禁止生成6个独立大HTML。**Phase 2/3 必须基于 Phase 1 产物加工，禁止重新切分原始 HTML 或重新生成框架。** | 产品不可用 |

## 1. 三阶段递进工作流程

> **递进原则**：三个阶段是**连续的加工链**，不是独立的平行工程。
> - **Phase 2 完全基于 Phase 1** 的产物进行精修：在 `metadata/科目_带锚点.html` 和 `六科_章节树.json` 的基础上，逐节插入语义锚点并修复标题层级。禁止重新生成框架或重新切分原始 HTML。
> - **Phase 3 完全基于 Phase 2** 的精修产物进行大白话生成：读取 `metadata/精修片段/科目/第X章/第X节.html` 或 `metadata/科目_带锚点_精修.html`，逐节产出右栏解读内容。禁止脱离已确认的精修锚点另起炉灶。

### 第一阶段：三栏阅读器框架（统一入口）

**交付物**：
- `index.html`：统一框架页面
- `app.js`：三栏交互+同步滚动+目录跳转+动态加载
- `style.css`：响应式样式
- GitHub Pages 部署完成

**技术规格**：
- **科目选择**：左上方显示2026年注册会计师考试科目完整名称（会计、审计、财务成本管理、公司战略与风险管理、经济法、税法），切换科目时整页刷新。
- **左栏**：章节目录，显示该科目**所有章和节（只到节）**，不显示更细层级。LLM必须读取原文件，把章节名字完整找到。点击某**章** → 中栏原文件跳到该章；点击某**节** → 中栏原文件跳到该节。
- **中栏**：加载该科目的**完整原教材HTML**（全本加载，不是一章章抽取），零改动，仅含锚点标记。文件格式为 `metadata/科目_带锚点.html`。
- **右栏**：大白话内容，根据中栏滚动位置同步加载对应内容。
- **分隔线**：中栏和右栏之间的分隔线可左右拖动调节宽度。
- **响应式**：必须同时适配手机、平板、电脑等各种终端。
- **加载方式**：根据用户选择的科目，动态请求该科目的 `_带锚点.html`（全本）和对应大白话数据。

**检查点**：
- 框架完成后上传GitHub，用户打开GitHub Pages链接验证交互是否正常。
- 用户确认后，压缩上下文，只记录"框架已完成"。

### 第二阶段：六科语义锚点 + 层级修复（LLM逐节精读）

> **基于**：第一阶段的 `metadata/科目_带锚点.html`（章-节锚点版）和 `metadata/六科_章节树.json`。

**Phase 2 核心任务**：
1. **语义锚点**：LLM 逐节阅读原文，根据**大白话生成单元的完整性**在节内插入不可见锚点。锚点不是为了对应标题层级，而是为了划定"两个锚点之间的原文 = 一篇大白话的输入"。大白话和原文需要同步滚动。
2. **层级修复**：修正 PDF 转 HTML 导致的标题标签层级混乱（如五级标题被误标为 h1），生成层级正确的 HTML。**层级修复只改标签，不改内容；语义锚点只插锚点，不改文字。两者是独立任务，不可混为一谈。**

**语义锚点黄金标准（经会计第1章第1节试点验证）**：

> **核心原则**：一个锚点 = 一篇大白话能自然讲完的语义单元。锚点密度必须让大白话篇幅落在 **300-800字** 的理想区间。

| 场景 | 处理方式 | 试点案例 |
|------|----------|----------|
| **并列论述，主题统一** | 合并为 **1个锚点**。多个并列点讲同一主题时，整体解读逻辑更顺畅。 | "会计的作用"（第一/第二/第三）合并为1个锚点 |
| **特定群体/独立主题** | 单独设锚点。即使前面有同级标题，只要对象或主题发生明显转换，就独立成篇。 | "注册会计师职业道德"从"会计人员职业道德"中独立 |
| **同属体系但各自太薄** | 合并为 **1个锚点**。避免一两句话就设一个锚点，导致大白话无话可讲。 | "具体准则"+"解释"+"应用指南"+"小企业会计准则"合并为"会计准则体系其他组成部分" |
| **同属责任/制度体系的多小点** | 合并为 **1个锚点**。多个小点共同构成一个完整制度/责任框架时，按"谁负责什么"统一讲解。 | "会计工作责任"1-5点合并为1个锚点 |
| **核心概念，内容极长** | 必须**单独设锚点**。篇幅超过800字的核心概念，单独成篇才能保证解读深度。 | "基本准则"单独设锚点（约700字） |
| **表格/例题/公式** | 必须完整地落在**同一个锚点边界内**，禁止被锚点切断。 |

**锚点命名规范**：
- 格式：`科目-第X章-第X节-语义标签`
- 语义标签要简短、直白（如`-会计工作责任`、`-基本准则`）
- 禁止过度嵌套层级（如`-会计工作责任-主体责任与人员责任`这种过长命名不再使用）

**工作单元**：**一科一节**（不是一章）。六科合计约 500+ 节，必须细水长流，防止上下文爆炸。

**工作流程（按科→节推进）**：

```
Step 1: 提取该节原始 HTML（基于现有章-节锚点边界）
Step 2: LLM 完整阅读该节内容
Step 3: LLM 输出该节的「精修 HTML 片段」
        a. 层级修复：校正该节内部的标题标签层级（h1→h2/h3/h4/h5）
        b. 文字格式修复：修正 PDF 转 HTML 导致的明显错误（如序号缺失、错行）
        c. 语义锚点：根据"大白话生成单元"的完整性插入锚点，
           格式：<div id="科目-第X章-第X节-语义标签" class="section-anchor"></div>
        d. 除此之外，原文文字、顺序、表格结构零改动
Step 4: 保存该节精修片段到 `metadata/精修片段/科目/第X章/第X节.html`
Step 5: 同时替换合并到 `metadata/科目_带锚点.html`（保证线上实时可预览）
Step 6: 提交 GitHub
Step 7: 用户检查确认
        - 打开 GitHub Pages 查看原文渲染效果
        - 确认语义锚点密度是否合适（大白话单元的粒度）
        - 确认标题层级修复是否正确
        - 如有问题，返回 Step 3 调整
Step 8: 压缩上下文（主上下文只保留一句话："[日期] [科目] 第X章第X节精修完成，用户已确认。"）
Step 9: 下一节
```

**全科合并流程（单科所有节确认完成后执行一次）**：

```
Step A: 将本科所有已确认的精修片段按顺序合并
Step B: 生成 `metadata/科目_带锚点_精修.html`（全本，替换原 `_带锚点.html`）
Step C: 生成 `metadata/科目_语义锚点映射.json`（记录所有锚点 ID → 章节位置 → 语义标签）
Step D: 更新 `app.js` 加载精修版（或直接覆盖原文件）
Step E: 用户全科浏览确认
Step F: 压缩上下文（主上下文只保留"[日期] [科目] Phase 2 完成，全本精修已合并。"）
```

**交付物**：
- 过程中：`metadata/精修片段/科目/第X章/第X节.html`（每节一个）
- 单科完成后：`metadata/科目_带锚点_精修.html` + `metadata/科目_语义锚点映射.json`

**检查点**：
- **每节确认**：用户必须逐节确认，未经确认不得进入下一节。
- **单科确认**：本科全部节合并后，用户在三栏阅读器中整体浏览一遍，确认锚点跳转和层级渲染无误。
- **上下文压缩**：用户说出"确认"、"继续"、"下一节"后，必须立即压缩上下文，详细过程写入 `ITERATION_LOG_v2.md`。

### Phase 2 经验教训与自动化检查清单（经会计第1-4章验证）

> **本清单为Phase 2每次完成一节后必须执行的强制质检流程。**

#### ✅ 成功经验

1. **h3和h4标题必须100%覆盖语义锚点**
   - 所有以 `一、` `二、` `三、` 等开头的小节标题（h3），前面必须紧跟一个语义锚点。
   - 所有以 `（一）`、`（二）`、`（三）` 等开头的子小节标题（h4），前面**同样必须紧跟一个语义锚点**。
   - **遗漏h3锚点会导致目录无法跳转；遗漏h4锚点会导致同级内容无法精准定位，用户体验断裂。**

2. **层级修复的完整范围**
   - 不仅要把异常 `h1` 修复为 `h2/h3/h4/h5`，还要检查**开闭标签是否匹配**（如 `<h4>...< /h3>` 这种错误）。
   - h4 以下（（一）、1.、①）的层级必须一致，不能出现 h3 下面突然跳回 h1 的情况。

3. **例题步骤必须统一为 `<p>（n）</p>` 格式**
   - PDF转HTML时常把例题步骤转成 `<ol start="n"><li>...</li></ol>`，这会破坏语义锚点边界和移动端排版。
   - 修复方法：将所有 `<ol>` 例题步骤替换为 `<p>（1）...</p>` 纯段落格式，保持与原文序号一致。

4. **半角括号 `(n）` 必须全局清除**
   - PDF转换常见的半角左括号+全角右括号组合 `(1）` `(2）` 等，必须批量替换为全角 `（1）` `（2）`。

5. **小节的语义标签必须简短、直白**
   - 命名紧贴h3标题的核心名词，如 `-存货的初始计量`、`-固定资产的后续支出`。
   - 禁止过长嵌套命名（如 `-会计工作责任-主体责任与人员责任`）。

#### ❌ 失败教训（严禁再犯）

1. **上下文过满导致跳过用户确认**
   - 会计第4章曾因连续处理多节未压缩，导致LLM上下文饱和，进而未经用户确认就推进到下一节，造成返工。
   - **铁律**：每完成一节，必须等待用户确认，确认后立即压缩上下文，无论当前状态看起来多么"正常"。

2. **只凭肉眼检查导致遗漏系统性错误**
   - 早期仅靠LLM肉眼扫读，遗漏了5个缺锚点的h3标题、6个半角括号、多处ol乱用。
   - **铁律**：每节完成后必须用脚本自动化检查（见下文检查脚本），不能仅靠LLM肉眼。

3. **h4锚点被系统性遗漏（会计第1-6章审计发现）**
   - 早期仅要求h3标题加锚点，导致第1-6章累计遗漏47个h4锚点。用户在手机上浏览时发现大量（一）（二）（三）内容无法精准跳转。
   - **铁律**：h3和h4标题前面必须100%覆盖语义锚点，缺一不可。

4. **标题层级修复不彻底引发连锁问题**
   - 第1章第4节 "一、会计要素定义及其确认条件" 原为 `<h1>` 开标签配 `</h3>` 闭标签，早期只修了开标签，导致渲染异常。
   - **铁律**：层级修复必须同时检查 `开标签级别 == 闭标签级别 == 实际语义级别`。

5. **批量修复脚本误判章标题（会计第8-9章）**
   - 会计第8-9章的章标题 `<h1>第八章 负债</h1>` 和 `<h1>第九章 职工薪酬</h1>` 在批量处理异常h1时被脚本误降级为 `<p>`，导致目录中缺失章标题层级，渲染异常。后续由LLM检查才发现并修复。
   - **铁律**：批量修复脚本必须设置章标题白名单（`第X章` 正则匹配），严禁将章标题纳入异常h1处理范围。

6. **大章节ol残留量大，手动处理不可行（会计第13章）**
   - 会计第13章（金融工具与保险合同）长达18.4万字符，含有88处 `<ol>` 残留，手动逐一替换效率极低且易遗漏。最终通过脚本批量提取 `<li>` 内容并自动转换为 `<p>（n）</p>` 格式才完成清理。
   - **铁律**：当单章ol残留超过5处时，必须使用自动化批量替换脚本，禁止手动逐个处理。

7. **h4下的`<p>（n）...`子标题被系统性遗漏为h5（会计第13章）**
   - 会计第13章中存在大量h4标题下以 `<p>（1）...`、`<p>（2）...` 形式出现的子主题，它们本应和已有的h5标题并列，但被PDF转HTML错误降级为普通段落。例如 `（2）将混合合同指定为以公允价值计量且其变动计入当期损益的金融工具` 与 `h5: 1. 嵌入衍生工具的分拆` 是并列关系，却仍是 `<p>`。
   - **判断标准**：位于h4下、以 `（n）` 开头、文本较短（<50字符）、后面跟着大量解释内容（>400字符），且与同h4下的其他h5呈并列结构。
   - **铁律**：大章节处理完成后，必须用脚本扫描h4下的所有 `<p>（n）短文本</p>`，将符合标题特征的项批量提升为h5。

8. **PDF转HTML导致的断行分裂需批量合并（会计第13章）**
   - PDF转HTML时常将一行文字在关键词处（如`以及`、`除`、`或`、`一`、`项`、`形`、`如`、`资`、`产`）错误截断，分成两个 `<p>` 标签。例如：`计量的一` / `项或一组金融资产`、`风险敞口及其形` / `成原因`、`<p>（3）相关金融资产...属于衍生工具（...以及</p>` / `<p>被指定为...除外）</p>`。
   - **铁律**：处理完成后必须用正则扫描上一 `<p>` 以截断词结尾、下一 `<p>` 明显接着同一句话的断行分裂，并批量合并为单个 `<p>`。

#### 🔧 强制检查脚本（每节/每章完成后执行）

```python
import re

# 0. 检查章标题必须是 h1（2026-04-15新增，防止批量脚本误降级）
h1s = re.findall(r'<h1[^>]*>.*?</h1>', html, re.DOTALL)
assert len(h1s) >= 1, "章标题缺失"
for h in h1s:
    text = re.sub(r'<[^>]+>', '', h).strip()
    if re.search(r'第[一二三四五六七八九十]+章', text):
        assert h.startswith('<h1'), f"章标题被错误降级: {text}"

# 1. 检查所有 h3 标题前面是否有语义锚点
missing_h3 = []
for m in re.finditer(r'<h3 id="[^"]*">([一二三四五六七八九十]+[、．.][^<]+)</h3>', html):
    before = html[max(0, m.start()-200):m.start()]
    if 'section-anchor' not in before:
        missing_h3.append(m.group(1))
assert len(missing_h3) == 0, f"缺锚点的h3: {missing_h3}"

# 2. 检查所有 h4 标题前面是否有语义锚点
missing_h4 = []
for m in re.finditer(r'<h4 id="[^"]*">（[一二三四五六七八九十]+）[、．.]?[^<]*</h4>', html):
    before = html[max(0, m.start()-120):m.start()]
    if 'section-anchor' not in before:
        missing_h4.append(m.group(0))
assert len(missing_h4) == 0, f"缺锚点的h4: {missing_h4}"

# 3. 检查标题开闭标签是否匹配
for m in re.finditer(r'(<h[1-5][^>]*>)(.*?)(</h[1-5]>)', html, re.DOTALL):
    open_level = m.group(1)[2]
    close_level = m.group(3)[3]
    assert open_level == close_level, f"层级不匹配: {m.group(0)[:50]}"

# 4. 检查半角括号
bad_parens = re.findall(r'\([0-9]+[）\)]', html)
assert len(bad_parens) == 0, f"半角括号残留: {bad_parens}"

# 5. 检查 <ol> 乱用（例题步骤应改为 <p>（n）</p>）
ol_tags = re.findall(r'<ol[^>]*>', html)
assert len(ol_tags) == 0, f"ol 残留: {len(ol_tags)} 处"

# 6. 检查h4下是否有遗漏的<h5>子标题（2026-04-16新增）
# 当h4区块内同时存在h5和<p>（n）短文本时，短文本应提升为h5
h4_positions = []
for m in re.finditer(r'<h4[^>]*>', html):
    h4_positions.append(m.end())
for i, h4_end in enumerate(h4_positions):
```

> **基于**：第二阶段精修后的 `metadata/精修片段/科目/第X章/第X节.html` 或合并后的 `metadata/科目_带锚点_精修.html`。

**黄金Prompt标准**：
1. 严格基于原文进行解读
2. 禁止根据标题自行发挥，禁止网络摘取
3. 专业术语第一次出现用括号标注通俗解释
4. 能用数字举例的必须举例，能用简单类比的必须使用
5. 必须解释"为什么"，不能只陈述"是什么"
6. 大白话篇幅约为原文的70%-150%（原文短时扩展，原文长时保持精简）
7. 中间不要空行空段，保持连贯叙述流
8. 语言口语化、有节奏感，像老师在给学生讲课
9. **必须覆盖**：表格逐行解释、例题考点拆解、公式逻辑解释、分录借贷含义
10. **禁止出现**通用套话："保证信息的真实性和可靠性"、"提高信息的可比性"、"与国际惯例接轨"、"防范风险和保护利益相关者"、"资本市场对外开放的必要条件"

- `output/科目/XXX.md`

### Phase 3 第1章经验教训（2026-04-19经会计第1章62篇验证）

> **本清单为Phase 3每次完成一章后必须执行的强制质检流程。**

#### ✅ 成功经验

1. **四步结构优于旧版**
   - `## 这段到底在讲什么？` 作为固定开篇标题，统一了风格，小白更容易跟随
   - `## 你可能会卡在这里` 用自然引导语（"你可能会问""这里容易晕"）代替编号列表，更像真人讲课
   - `## 逐一破解` 按 why→what→how 逻辑链条展开，不是教材翻译

2. **自检报告在文件末尾**
   - 用 `---` 分隔线与正文隔开，不污染正文
   - 包含7项检查，生成时逐条核对，不通过不提交

3. **人工处理长文+子代理处理短文的混合策略可行**
   - 长文（>1000字原文）人工处理，质量可控
   - 短文（<500字原文）子代理批量，效率提升
   - 关键是必须有**后置质量门禁**拦住子代理的劣质输出

#### ❌ 失败教训（严禁再犯）

1. **Markdown格式未渲染（技术事故）**
   - 根因：`app.js` 第363行 `escapeHtml(content).replace(/\n/g, '<br>')` 把 `##` 和 `**` 都当纯文本转义了
   - 后果：网页上显示 `##这段到底在讲什么？` 和 `**粗体**`，格式完全失效
   - **铁律**：大白话内容使用 Markdown 语法，前端必须提供 `mdToHtml()` 解析器，不能用 `escapeHtml()`
   - 修复：添加 `mdToHtml()` 函数，支持 `##`→`<h3>`、`###`→`<h4>`、`**`→`<strong>`、段落自动分段

2. **中文数字多次出现（质量问题，已纠正两次）**
   - 违规案例："算出来现值六千二百七十三万，低于账面价值，得确认二百二十七万减值损失"
   - 正确写法："算出来现值6,273万，低于账面价值，得确认227万减值损失"
   - **铁律**：所有金额、数量、百分比必须使用阿拉伯数字。prompt中必须明确写入"中文数字零容忍"
   - 防护：merge脚本中集成 `cn2an` 自动转换 + 扫描脚本事后检查

3. **非结构性加粗泛滥（子代理问题）**
   - 子代理在正文中滥用 `**加粗**` 来强调短语，违反 Rule #5
   - **铁律**：`**` 只能用于结构性标记（条件翻译/思路拆解/数字推导/陷阱提示/表格全貌/逐行解读/纵向逻辑/借/贷/例编号）
   - 防护：merge脚本中加 `clean_bold()` 函数，自动删除非结构性加粗

4. **阶段标记泄漏到正文（子代理问题）**
   - 子代理把思考过程（"阶段一""难点扫描""具象化方案"）写入正文
   - **铁律**：prompt中必须明确"思考过程只在你脑子里完成，严禁输出到文件"
   - 防护：merge脚本中加 `clean_stages()` 函数 + 事后扫描脚本检查

5. **虚构人物编故事（子代理问题）**
   - 子代理编"老王借钱""小李炒股"等虚构人物故事
   - **铁律**：比喻必须基于原文场景，用原文中的主体（企业/投资者/管理层）来讲解
   - 防护：prompt中明确禁止虚构人物 + 扫描脚本检查关键词（老王/小李/小张/小明/小红）

6. **篇幅不足（个别案例）**
   - 部分短文大白话字数刚好踩线（80%），内容不够深入
   - **铁律**：≥80%是底线不是目标，质量合格的大白话应该≥100%
   - 防护：自检报告中必须精确计算字数比例，低于90%标红警告

7. **子代理 vs 人工的权衡**
   - 用户明确要求"不要子代理，要我亲自完成"
   - 实际执行中，62篇全部人工需要约60轮工具调用，超出预算
   - 最终方案：30篇人工 + 31篇子代理，子代理输出经质量门禁过滤
   - **教训**：对于用户明确禁止子代理的场景，必须提前规划人力分配，不能到最后被迫使用子代理

### 3.1 Push 验证铁律（2026-04-18新增，经第5章部署事故验证）

**问题**：本地 commit 后未执行 `git push`，导致 GitHub Pages 持续展示旧版本，用户误以为工作未完成。

**强制规则**：
1. 每次 commit 后必须执行 `git push origin main`
2. push 后立即执行 `git status` 验证 `"Your branch is up to date with 'origin/main'"`
3. 若状态显示 `ahead of 'origin/main'`，禁止汇报"已完成"，必须先 push
4. 用户反馈"网站上没看到"时，第一条排查指令永远是检查 `git status`

## 5. 结构级质量审计规范（2026-04-16新增）

> 基于会计第19-22章修复实战，对PDF转HTML导致的系统性结构问题建立检测和修复标准。

### 5.1 例题编号错乱审计

**问题定义**：PDF转HTML时，例题 `[例XX-XX]` 内部的 `（1）（2）（3）...` 编号经常出现重复、跳跃、缺失。

**检测脚本标准**：
```python
import re
examples = re.finditer(r'\[例(\d{2}-\d+)\](.*?)(?=\[例|\n<h[1-6]|</div><div id="会计-第)', block, re.DOTALL)
for m in examples:
    ex_id = m.group(1)
    nums = re.findall(r'<p>（([0-9]+)）', m.group(2))
    int_nums = [int(n) for n in nums]
    for i in range(1, len(int_nums)):
        if int_nums[i] == 1 and int_nums[i-1] > 1:
            flag(f'[例{ex_id}] 编号重复重置: {int_nums}')
        if int_nums[i] > int_nums[i-1] + 1:
            flag(f'[例{ex_id}] 编号跳跃: {int_nums}')
```

**修复决策树**：
1. **先由LLM阅读原文**：判断这些 `（n）` 是否属于**同一例题内部的连续步骤**。
2. 如果是同一例题步骤 → **强制修复为连续编号**（1,2,3...）。
3. 如果是**层级嵌套**（如 h4 原则下的 `（1）` 与例题分析的 `（1）` 之间）→ **保持原文结构，不修改编号**。
4. **严禁机械统一**：脚本只能标红，修复动作必须LLM确认。

### 5.2 会计分录 MathML 规范化

**问题定义**：PDF转HTML时，会计分录中的金额和计算式被包裹在 `<math display="block">` 中，导致数字和运算符被垂直堆叠显示为乱码（如 `[2 0 0 0 0 × (6. 4 5 - 6. 2)] 5 0 0 0`）。

**判定标准**：满足以下任一条件，该 `math` 标签必须被替换为纯文本：
1. 位于 `<p>借：xxx</p>` 或 `<p>贷：xxx</p>` 的**下一段落**中
2. annotation 文本中包含 `借`、`贷`、`×`、`%`、`[数字]` 等会计计算符号
3. `display="block"` 且前后文为会计分录结构

**修复标准格式**：
```html
<!-- 错误 -->
<p>借：长期借款——美元</p>
<p><math display="block">[20000×(6.45-6.2)] 5000</math></p>
<p>贷：在建工程</p>
<p>5000</p>

<!-- 正确 -->
<p>借：长期借款——美元 [20000×(6.45-6.2)] 5000</p>
<p>贷：在建工程 5000</p>
```

**强制要求**：必须扫描并清零 block math 在会计分录中的使用。

### 5.3 h5/p 边界判定规则

| 元素 | 使用场景 | 禁用场景 |
|------|---------|---------|
| `<h5>` | 章-节-小节下的**主题分段标题**（如"1. 以公允价值为基础计量的"） | 例题内部的 `（n）步骤说明` |
| `<p>` | 例题内部的步骤说明、分录、分析文字 | 作为主题分段标题使用 |

**检测脚本**：
```python
# h5误升级检查
for m in re.finditer(r'<h5[^>]*>（[0-9]+）[）)．.\s]*[^<]{3,40}</h5>', block):
    before = block[max(0, m.start()-200):m.start()]
    if '[例' in before or '账务处理如下' in before or '会计分录' in before:
        flag('h5误升级：例题步骤被提升为h5')
```

### 5.4 断行分裂（断句）扫描规则

**升级检测逻辑**：
```python
def find_splits(block):
    pattern = r'<p>([^<]*?)(以及|除|或|一|项|形|如|资|产|工|具|可|的|纳|差|以|计|定|准|理)</p>\n<p>([^<]{2,})</p>'
    for m in re.finditer(pattern, block):
        first = re.sub(r'<[^>]+>', '', m.group(1)).strip()
        sep = m.group(2)
        second = re.sub(r'<[^>]+>', '', m.group(3)).strip()
        if re.match(r'^(借|贷)[:：]', first) and re.match(r'^\d', second):
            continue
        if re.match(r'^(例如|如果|然而|此外|同时|因此|但是|需要注意的是|一是|二是|首先|其次|再次|最后)', second):
            continue
        if first.endswith(('。', '：', '；', '，', '、', '”')):
            continue
        if sep == '的' and not second.startswith(('是', '确', '原', '基', '结', '情', '价', '成')):
            yield m
        elif sep in ['可', '以', '除', '或', '项', '形', '如', '资', '产', '工', '具', '纳', '差', '计', '定', '准', '理']:
            yield m
```

**修复原则**：LLM阅读原文语义，确认两句实为一句被截断后，方可合并。

### 5.7 会计分录纯文本格式规范（2026-04-16第三次升级）

> 基于1-22章237个例题全覆盖审计，建立会计分录的HTML表示标准。

**问题定义**：PDF转HTML后，会计分录经常出现以下格式异常：
1. **`<br />` 连接多行分录**：多个借/贷条目被塞在同一个 `<p>` 标签内，用 `<br />` 换行
2. **混合分录**：借方和贷方被合并到同一个 `<p>` 标签内（如 `<p>借：合同资产 5351608 贷：财务费用 5351608</p>`）
3. **Inline Math 跨越贷/借边界**：math 标签的结束标签与"贷"字连在一起，导致格式错乱
4. **科目与金额分离**：已被 5.2 覆盖的 block math 问题，以及金额被拆到下一 `<p>` 的问题

**格式标准（强制）**：
```html
<!-- 正确 -->
<p>借：银行存款 77 500</p>
<p>资本公积——其他资本公积 279 000</p>
<p>贷：股本 15 500</p>
<p>资本公积——股本溢价 341 000</p>

<!-- 错误：br连接 -->
<p>借：其他业务成本 160000<br />\n贷：投资性房地产累计折旧 160000</p>

<!-- 错误：混合分录 -->
<p>借：合同资产 5351608 贷：财务费用、利息收入等 5351608</p>
```

**判定与修复**：
1. `<br />` 在分录中：**必须拆分为独立 `<p>`**（每个借/贷/中间科目一行）
2. 同一 `<p>` 内同时出现 `借[:：]` 和 `贷[:：]`：**必须拆分为独立 `<p>`**
3. math 标签与"贷"/"借"字粘连：检查并修复 HTML 结构
4. **MathML inline 导致混合分录**：math 标签横跨借/贷边界，或金额数字与 `贷：` 字粘连在同一 `<p>` 内。必须将 math 提取为纯文本金额，并将借、贷拆分为独立 `<p>`。
5. 分录条目（借/贷/中间科目）与金额被拆到不同 `<p>`：**必须合并到同一 `<p>`**
6. **借/贷方金额完全缺失**：`<p>借：科目</p>` 后直接跟 `<p>贷：科目 金额</p>`，借方行无金额数字。应**由 LLM 根据例题上下文或下一贷方金额判断缺失金额并补全**。
7. **金额说明前置拆行**：借方/贷方金额被拆成说明行+数字行，如 `<p>借：营业成本</p><p>(3500×30%) 1050③</p>`。应将说明括号并入同一 `<p>`，或删除说明只保留金额（视原文语义而定）。
8. 表格化分录：`<table>` 包裹的会计分录必须逐条由 LLM 判断后转为 `<p>` 格式（详见 5.10）

**检测脚本**：
```python
# 检查 br 在分录中的残留
for m in re.finditer(r'<p>(.*?)</p>', block, re.DOTALL):
    content = m.group(1)
    if '<br />' in content and ('借' in content or '贷' in content):
        flag(f'<br />分录残留: {content[:80]}')

# 检查混合分录
for m in re.finditer(r'<p>(.*?)</p>', block, re.DOTALL):
    text = re.sub(r'<[^>]+>', '', m.group(1))
    if '借' in text and '贷' in text and '\n' not in m.group(1):
        if re.search(r'借[:：]', text) and re.search(r'贷[:：]', text):
            flag(f'混合分录残留: {text[:80]}')

# 检查金额单独成行（排除贷方后面跟说明文字的正常情况）
for m in re.finditer(r'<p>借[:：]([^<]+)</p>\n<p>(\d[^<]*)</p>', block):
    flag(f'借方金额分行: {m.group(1)} -> {m.group(2)}')
for m in re.finditer(r'<p>贷[:：]([^<]+)</p>\n<p>(\d[^<]*)</p>', block):
    if re.match(r'^\d', m.group(2)) and not re.search(r'年|月|日|分析|处理', m.group(2)):
        flag(f'贷方金额分行: {m.group(1)} -> {m.group(2)}')

# 检查金额完全缺失（借/贷行无数字，但下一段是对方科目+金额）
for m in re.finditer(r'<p>(借[:：][^\d<]{1,30})</p>\n<p>(贷[:：][^<]*\d[^<]*)</p>', block):
    flag(f'借方金额缺失: {m.group(1)} 后接 {m.group(2)[:60]}')
for m in re.finditer(r'<p>(贷[:：][^\d<]{1,30})</p>\n<p>(借[:：][^<]*\d[^<]*)</p>', block):
    flag(f'贷方金额缺失: {m.group(1)} 后接 {m.group(2)[:60]}')

# 检查金额说明前置拆行（公式/括号在单独一行）
for m in re.finditer(r'<p>([借贷][:：][^<]{1,20})</p>\n<p>\([^<]*\)\s*\d[^<]*</p>', block):
    flag(f'金额说明拆行: {m.group(1)}')
```

### 5.8 表格结构修复规范（2026-04-16新增）

**问题定义**：PDF转HTML时，跨页表格常被拆分为多个 `<table>`，且后续 table 中重复插入表头行。同时表格标题和单位说明常被拆分为多个 `<p>` 标签。

**判定标准**（满足以下全部条件时，必须合并）：
1. 同一例题/段落内，出现 2 个及以上连续 `<table>`
2. table 之间仅有短分组标签（如 `<p>第一组</p>`、`<p>第X年</p>`）
3. 非首 table 中包含与首 table 相同的表头结构（`rowspan`/`colspan` 特征一致）

**修复标准**：
1. 将多个 `<table>` 合并为 1 个
2. 删除重复的表头 `<tr>` 行
3. 将分组标签移入 table 内，作为普通数据行（如 `<tr><td colspan="N">第一组</td></tr>`）
4. 表格标题和单位说明必须合并为一个 `<p>`（如 `<p>表7-1 单位：万元</p>`）

**检测脚本**：
```python
# 检查表格标题分离
title_splits = re.finditer(r'<p>(表\d+-\d+[^<]*)</p>\n<p>([^<]+)</p>\n<table>', block)
for m in title_splits:
    flag(f'表格标题分离: {m.group(1)} + {m.group(2)}')

# 检查表格拆分
tables = list(re.finditer(r'<table>.*?</table>', block, re.DOTALL))
for i in range(len(tables)-1):
    between = block[tables[i].end():tables[i+1].start()]
    between_text = re.sub(r'<[^>]+>', '', between).strip()
    if len(between_text) < 50 and between.count('<p>') <= 2:
        table2 = tables[i+1].group(0)
        if 'rowspan' in table2 or 'colspan' in table2 or '<th>' in table2:
            flag(f'表格可能拆分: 第{i+1}与{i+2}个table之间="{between_text}"')
```

### 5.9 表格 CSS 渲染规范（2026-04-17新增）

> 基于表9-1/表9-2（含 rowspan/colspan 的复杂表格）渲染异常修复经验。

**问题定义**：
1. **`table-layout: fixed` 对复杂表格是灾难性的**：对 `tbody`/`tr` 强制 `display: table; width: 100%; table-layout: fixed` 会导致浏览器对有 `rowspan`/`colspan` 的表格进行**等宽列分配**，使跨行单元格（如"柜式空调生产车间"）被严重挤压，内容堆叠、边框错位。
2. **`display: block` + `overflow-x: auto` 直接作用于 `<table>`**：会破坏表格的正常流式布局算法，导致 rowspan 渲染异常。
3. **单元格文字溢出**：部分表头或数据单元格内容过长（如公式说明 `②=期初③×7.82%`），超出单元格后不可读。

**CSS 规范（强制）**：
```css
/* 横向滚动应由原文容器提供，而非 table 自身 */
.original-content {
  overflow-x: auto;  /* 允许超宽内容横向滚动 */
}

/* table 保持原生 display: table，不强制 fixed layout */
.original-content table {
  border-collapse: collapse;
  margin: 12px 0;
}

/* 数据单元格允许自动换行，防止溢出 */
.original-content td {
  word-break: break-all;
}

/* 表头尽量保持语义完整 */
.original-content th {
  word-break: keep-all;
}
```

### 5.10 表格化分录识别与转换规范（2026-04-17新增）

> 基于全书 10 个被 `<table>` 错误包裹的会计分录修复经验。

**问题定义**：PDF转HTML时，部分会计分录被转换为两列 `<table>`（左列科目、右列金额），丢失了分录的段落语义，且在大白话同步滚动和格式渲染上均不正常。

**LLM 识别标准**（同时满足）：
1. `<table>` 内同时包含"借"和"贷"
2. 无正常表头（如"日期"、"项目"、"年份"、"科目名称"、"逾期情况"等）
3. 行数 ≤ 30（超长表格通常为正常数据表）

**转换规则**：
1. **逐条精确替换**：不得用正则批量替换所有含借/贷的 table，必须提取完整 HTML，人工（LLM）确认后再替换。
2. **保留续行缩进语义**：原表格中以 `——` 开头的续行（如 `——利息调整 50 501`）在转为 `<p>` 后应保持 `——` 前缀，以维持视觉层级。
3. **补全缺失的借贷前缀**：表格中部分中间行可能缺少"借："或"贷："前缀（因为原 table 将前缀放在首行），转换后不应盲目补全，应保持原文空格缩进习惯；但如果上下文明确属于借方/贷方续行，可保持原样。
4. **非分录文字单独成行**：表格中夹杂的说明文字（如 `"实际支付利息时："`、`"同时或资产负债表日,账务处理如下:"`）应转为普通 `<p>`，不包入分录行。
5. **大分录分段处理**：超过 20 行的超大分录表格，转换后应按原有空行/说明文字自然分段。

**检测脚本**：
```python
journal_tables = []
for m in re.finditer(r'<table>.*?</table>', block, re.DOTALL):
    table_html = m.group(0)
    text = re.sub(r'<[^>]+>', '', table_html).strip()
    has_header = any(k in text for k in ['日期', '项目', '说明', '摘要', '单位', 
        '期初', '期末', '年份', '股东', '逾期情况', '科目名称', '期初摊余成本'])
    is_journal = '借' in text and '贷' in text and not has_header
    rows = re.findall(r'<tr>(.*?)</tr>', table_html, re.DOTALL)
    if is_journal and len(rows) <= 30:
        journal_tables.append((m.start(), len(rows), table_html))
        flag(f'疑似表格化分录: 位置={m.start()} 行数={len(rows)}')
```

## 6. 大白话风格规范（Phase 3 执行标准）

> **适用范围**：所有科目、所有章节的大白话生成。
> **执行方式**：LLM按此规范生成，人只审核不写作。

### 6.1 去教材味（零容忍）

- ❌ 教材编号：一、二、三 / （一）（二）（三）/ 1. 2. 3.
- ❌ 加粗子标题：**xxx**
- ❌ 结构化套词：首先、其次、最后 / 一是…二是… / 第一、第二
- ❌ 教材书面语：上述、主要规范、值得注意的是、需要指出的是、综上所述
- ❌ AI套话：**讲真**、**说白了**、我们不难发现、不难发现、值得注意的是

### 6.2 真正的口语化

- ✅ 自然的说话节奏，像跟熟人解释事情
- ✅ 不用任何"伪口语"口头禅，每篇最多1个语气过渡
- ✅ 长短句交错，有停顿感，不是一气读完的书面语
- ✅ 允许偶尔的吐槽、提醒、反问，但要自然不刻意

### 6.3 具象化比喻（不编故事）

- ❌ 禁止为了比喻而编造一个完整的故事情节
- ❌ 禁止比喻与原文内容完全无关
- ✅ 比喻必须基于原文场景自然产生
- ✅ 可以用原文本身就有的关系、场景做类比
- ✅ 比喻要简短，一两句话点透，不展开编故事

### 6.4 打破教材顺序

- ❌ 不能顺着教材段落逐句翻译
- ✅ 按"先讲为什么→再讲是什么→最后讲怎么做"重组
- ✅ 把最难懂的概念放在有上下文铺垫之后讲

### 6.5 分录格式

- ✅ 借在上、贷缩进、金额右对齐
- ✅ 标准中国借贷格式，禁止英文科目

### 6.6 英文零容忍

- ❌ 任何英文词汇混入中文解读

### 6.7 风格统一

- 1-6章 + 后续全部章节，**同一个风格**
- 不能这篇像讲课、那篇像笔记
- 每篇按同一套规范生成

### 6.8 逐篇精修

- ✅ 每篇根据原文单独列出禁用词组
- ✅ 每篇结尾有自然收尾，不要"综上所述"
- ✅ 主代理抽查每批后半篇（子代理容易偷懒）

### 6.9 原文渲染检查（前端修复规范）

> PDF转HTML常见格式问题，必须在app.js中做前端自动修复，禁止修改原文HTML。

| 问题类型 | 检测方法 | 修复方式 | 本日案例 |
|---------|---------|---------|---------|
| 分录table化 | 扫描`<table>`中含"借/贷" | 转为`<pre>`标准分录格式 | 36个 |
| 数据table无样式 | 扫描无class/border的table | style.css添加table样式 | 56个 |
| LaTeX公式未渲染 | 扫描`\(...\)`格式 | 替换为可读HTML+CSS样式 | 1处 |
| 半角括号混用 | 扫描中文后接`()` | 自动转为全角括号 | 19处 |
| 半角冒号混用 | 扫描"借:/贷:" | 自动转为全角"借：/贷：" | 24处 |

**执行原则**：原文零改动，所有修复通过app.js前端JS在渲染时自动处理。

## 7. 异常处理规则

| 异常 | 处理方式 |
|------|---------|
| 用户发现质量问题 | 立即停止，分析原因，回滚到该章/节起点，重新生成 |
| LLM token额度告警 | 立即保存当前进度到 `PROGRESS_SNAPSHOT.json`，汇报并暂停 |
| 锚点划分有争议 | 暂停，提出2-3个方案供用户选择，不擅自决定 |
| 原文HTML有损坏/缺失 | 立即停止，汇报问题，等待用户修复原文 |
| 用户说"网站上没看到" | 第一条排查：`git status` 检查本地是否 ahead of origin |
| 用户指令与当前上下文不一致 | 立即确认，禁止假设执行 |
| 通信平台断连（Telegram/Discord等） | 立即检查代理配置（Clash端口7897）和网络状态，恢复后汇报中断期间的工作状态 |
