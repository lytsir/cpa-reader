## 0. 核心铁律（不可违反）

| 铁律 | 内容 | 违反后果 |
|------|------|---------|
| **原文零改动** | 中间栏加载的必须是完整原始HTML。切分只能插入不可见的 `section-anchor` 锚点。禁止修改原文文字、格式、表格、图片、公式。 | 项目失败，全部回滚 |
| **LLM亲自读原文** | 锚点划分和大白话生成必须由LLM逐段阅读原文内容后决定，不能仅依赖HTML标签（h1/h2）机械切分。 | 锚点错位，大白话脱离原文 |
| **逐章/逐节确认** | 每完成一章锚点或一节大白话，必须上传GitHub并由用户确认。未经确认不得进入下一步。 | 质量失控，返工量爆炸 |
| **上下文压缩** | 用户确认后，主上下文立即压缩，只保留完成记录。详细过程归档到日志文件，不得占用上下文空间。 | 上下文过满导致遗忘规则 |
| **统一框架+动态加载** | 最终产品为一个 `index.html` 三栏框架，各科原文和大白话作为外部资源按需加载。禁止生成6个独立大HTML。 | 产品不可用 |

## 1. 三阶段递进工作流程

### 第一阶段：三栏阅读器框架（统一入口）

**交付物**：
- `index.html`：统一框架页面
- `app.js`：三栏交互+同步滚动+目录跳转+动态加载
- `style.css`：响应式样式
- GitHub Pages 部署完成

**技术规格**：
- **科目选择**：左上方显示2026年注册会计师考试科目完整名称（会计、审计、财务成本管理、公司战略与风险管理、经济法、税法），切换科目时整页刷新。
- **左栏**：章节目录，显示该科目**所有章和节（只到节）**，不显示更细层级。LLM必须读取原文件，把章节名字完整找到。点击某**章** → 中栏原文件跳到该章；点击某**节** → 中栏原文件跳到该节。
- **中栏**：加载该科目的**完整原教材HTML**（全本加载，不是一章章抽取），零改动，仅含锚点标记。文件格式为 `metadata/科目_带锚点.html`。
- **右栏**：大白话内容，根据中栏滚动位置同步加载对应内容。
- **分隔线**：中栏和右栏之间的分隔线可左右拖动调节宽度。
- **响应式**：必须同时适配手机、平板、电脑等各种终端。
- **加载方式**：根据用户选择的科目，动态请求该科目的 `_带锚点.html`（全本）和对应大白话数据。

**检查点**：
- 框架完成后上传GitHub，用户打开GitHub Pages链接验证交互是否正常。
- 用户确认后，压缩上下文，只记录"框架已完成"。

---

### 第二阶段：六科锚点（LLM亲自读原文）

**工作流程（按科→章推进）**：

```
Step 1: LLM读取该章原教材HTML
Step 2: LLM人工识别语义单元（不只按h1/h2，要按知识点完整性）
        - 遇到表格/例题/公式必须整体保留在一个单元内
        - 遇到短标题/过渡句必须合并到相邻的实质内容单元
Step 3: 在该章各节末尾插入不可见锚点
        - 格式：<div id="科目-第X章-第X节" class="section-anchor"></div>
        - 禁止改动原文其他内容
Step 4: 生成该章锚点映射JSON（记录锚点ID→文件名/标题/位置）
Step 5: 上传GitHub（提交该章的 `_带锚点.html` 和 `_锚点映射.json`）
Step 6: 用户检查确认
        - 打开GitHub Pages查看
        - 验证锚点位置是否合理
        - 如有问题，返回Step 2调整
Step 7: 压缩上下文（主上下文只保留"XX科第X章锚点已完成"）
Step 8: 下一章
```

**交付物（每章）**：
- `metadata/科目_带锚点_第X章.html`
- `metadata/科目_第X章_锚点映射.json`

**检查点**：
- 单科全部章节的锚点完成后，用户在框架中整体浏览一遍，确认无误。
- 用户确认后，压缩上下文，只记录"XX科锚点已完成"。

### 第三阶段：大白话解读（逐节精修）

**工作流程（按科→章→节推进）**：

```
Step 1: 读取该节原文（HTML片段，从锚点之间提取）
Step 2: LLM按黄金Prompt逐篇生成大白话
        - 严格基于原文，禁止标题发挥
        - 禁止引入原文没有的知识点
        - 必须具象化解释"为什么"
        - 必须覆盖表格/例题/公式/分录
        - 语言口语化，像老师讲课
Step 3: 实质质检
        - 检查是否基于原文（比对关键术语）
        - 检查是否有通用套话/编造内容
        - 检查是否解释"为什么"
        - 不通过则返回Step 2重写
Step 4: 生成 .md 文件
Step 5: 上传GitHub
Step 6: 用户检查确认
        - 在三栏阅读器中查看该节效果
        - 确认大白话质量
Step 7: 压缩上下文（主上下文只保留"XX科第X章第X节大白话已完成"）
Step 8: 下一节
```

**黄金Prompt标准**：
1. 严格基于原文进行解读
2. 禁止根据标题自行发挥，禁止网络摘取
3. 专业术语第一次出现用括号标注通俗解释
4. 能用数字举例的必须举例，能用简单类比的必须使用
5. 必须解释"为什么"，不能只陈述"是什么"
6. 大白话篇幅约为原文的70%-150%（原文短时扩展，原文长时保持精简）
7. 中间不要空行空段，保持连贯叙述流
8. 语言口语化、有节奏感，像老师在给学生讲课
9. **必须覆盖**：表格逐行解释、例题考点拆解、公式逻辑解释、分录借贷含义
10. **禁止出现**通用套话："保证信息的真实性和可靠性"、"提高信息的可比性"、"与国际惯例接轨"、"防范风险和保护利益相关者"、"资本市场对外开放的必要条件"

**交付物（每节）**：
- `output/科目/XXX.md`

**检查点**：
- 每科第一章（约30-50节）全部完成后，用户确认风格统一再继续。
- 用户确认后，压缩上下文，只记录"XX科第1章大白话已完成"。

## 2. 质检标准（强制通过）

每篇大白话必须通过以下检查才能入库：

| 检查项 | 标准 | 未通过处理 |
|--------|------|-----------|
| 原文关联度 | 大白话中的核心概念必须能在原文中找到对应 | 立即重写 |
| 禁止套话 | 不得出现宪法列出的10条通用套话 | 立即重写 |
| 具象化 | 必须包含"比如/例如/相当于/就好比"等标志 | 返回补充 |
| 完整性 | 表格/例题/公式/分录必须被覆盖 | 返回补充 |
| 口语化 | 禁止公文腔、学术腔、摘要腔 | 立即重写 |
| 长度 | 大白话字数 ≥ 原文字数 × 0.5 | 返回扩展 |

## 3. GitHub同步规则

- **每次完成一章锚点** → 立即 `git add + commit + push`
- **每次完成一节大白话** → 立即 `git add + commit + push`
- **commit消息格式**：`[科目] 第X章锚点完成` 或 `[科目] 第X章第X节大白话完成`
- **GitHub Pages必须实时可访问**，用户随时能查看最新效果

## 4. 上下文压缩规则（防止再次过满）

**压缩触发条件**：用户说出"确认"、"没问题"、"继续"等通过指令后，立即执行。

**压缩动作**：
1. 将本章/本节的详细生成过程写入 `ITERATION_LOG_v2.md`
2. 主上下文中删除所有原文片段、生成内容、中间讨论
3. 主上下文中只保留一句话：`"[日期] [科目] [章节] 已完成，用户已确认。"`

**禁止行为**：
- 禁止将生成的9510篇文章内容塞入主上下文
- 禁止在上下文中保留超过3个已完成章节的详细过程

## 5. 异常处理规则

| 异常 | 处理方式 |
|------|---------|
| 用户发现质量问题 | 立即停止，分析原因，回滚到该章/节起点，重新生成 |
| LLM token额度告警 | 立即保存当前进度到 `PROGRESS_SNAPSHOT.json`，汇报并暂停 |
| 锚点划分有争议 | 暂停，提出2-3个方案供用户选择，不擅自决定 |
| 原文HTML有损坏/缺失 | 立即停止，汇报问题，等待用户修复原文 |
